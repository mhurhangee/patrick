#### Deferred minor items (logged, not fixed — revisit if they ever bite)
- `DocxEditorOverlays` **renders its own Sonner `<Toaster position="bottom-right" />`** (carried): if the host app also mounts a Sonner `Toaster` (Patrick does), there are two toast regions — toasts can double-render or land in the wrong corner. Fix is a product decision: either the editor owns its toasts (and the host must NOT mount one) or the editor drops the `<Toaster>` and relies on the host's. Left as a note — needs that decision + a host-integration smoke-test.
- `useHistory.push` **grouping keeps the intermediate state, not the pre-group one** (found 3g-bc): when rapid pushes fall within `groupingInterval`, the grouped branch rewrites the last undo entry's `state` to the *current* (intermediate) state rather than the original pre-group state, so an undo after a grouped burst lands on the intermediate value instead of jumping over the whole group. Also `defaultIsEqual` is `JSON.stringify`-based. Left alone: the editor uses `useDocumentHistory` (custom `isEqual`) and PM owns text undo, so the Document-level grouping path may not even be exercised — needs confirmation + a smoke-test before touching undo behavior.
- `useLayoutTriggers` **font `loadingdone` listener is document-global** (review #103, pre-existing): *any* web font finishing load — including Patrick's UI fonts (Hanken Grotesk/Lora) unrelated to the open doc — triggers a full `resetCanvasContext`+`clearAllCaches`+re-layout, causing a reflow/flicker. `FontFaceSet` doesn't say which font a doc uses, so scoping is non-trivial (e.g. track the doc's font families and filter the event). Deferred (optimization, not correctness).
- `useLayoutTriggers` **H/F re-layout effect re-fires on `runLayoutPipeline` identity churn** (review #103, pre-existing): the second effect lists `runLayoutPipeline` in deps, so unrelated edits that re-create that callback fire a redundant full layout pass. Fix = read it through a ref like the font effect now does, depending only on the H/F content props. Left untouched on purpose: it's an **H/F-adjacent** effect and H/F is deferred from this audit.
- `usePagesPointer.handlePagesClick` **possibly-unreachable read-only external-link branch** (carried): the `if (!surface)` path that `window.open`s an external link only runs when `activeSurface()` is null, but it returns `hiddenPMRef.current` which is non-null on any real click (HiddenProseMirror stays mounted even read-only). If genuinely dead, the read-only external-link-open intent is actually served by the `onHyperlinkClick` path instead. Verify against the hyperlink rework + a read-only-doc smoke-test before removing; harmless as-is.
- `usePagedScrollApi.scrollToParaIdImpl` **double flashPara** (carried): `flashPara()` fires once synchronously and again inside `runAfterPaint`, so an already-painted target flashes twice (~50ms apart — the highlight restarts). Benign; the two calls cover painted (immediate) + virtualized (post-paint) targets, and dropping either risks losing a flash case, so left as-is.
- **Visual-line nav sticky-X never resets on mouse-click / programmatic selection** (carried; lives in **core** `visualLineNavigation.handleVisualLineKeyDown`, now the single source after the 3e dedupe). `stickyX`/`lastVisualLineIndex` reset only on horizontal-arrow/typing keys — so after you click to a new caret position and press ArrowUp/Down, the move uses the *old* sticky X / line index and the caret jumps to the wrong column. Fix would validate that the tracked line still contains the caret `from` (else recompute line + reset sticky from the current position). Deferred: it's delicate caret-nav in **shared core** (affects the Vue adapter too) and needs interactive smoke-testing — not an unsupervised change.
- `useTableResizeState` **handle-drift past the min-width/height clamp** (carried): `handleMouseMoveUpdate` moves the visual resize handle by the raw pixel delta on every move, but only updates the committed twip width/height while within `MIN_*` bounds — so once you drag past the minimum the handle visually drifts off the column edge (the committed size stays correct). Fix would gate the visual move + start-coord reset on the same in-bounds check (all 3 modes). Deferred: it's cosmetic-during-drag and changes the feel of a working, smoke-tested gesture, which needs interactive validation rather than an unsupervised change.
- `ImageSelectionOverlay` scroll/resize **snap-back during an active resize**: a scroll or window-resize landing mid-resize briefly reads the un-committed DOM image rect into `overlayRect`, flickering the live preview; self-corrects on the next `mousemove`. A guard was tried and reverted (review #96) because the cure (a persistent gesture flag) was worse than the disease. Only worth revisiting if the flicker is ever actually observed.

- **Cutting `ContentControlWidgets` is a deeper (core-painter) change than expected (review #108).** You approved cutting the Word content-control / SDT form widgets, and removing the React `ContentControlWidgets` component is trivial — BUT the **core layout painter** (`docx-editor-core/.../layout-painter/sdtBoundary.ts`) independently paints the SDT trigger chrome (the clickable ☐/☒ checkbox, ▾ dropdown, 📅 date, ＋/✕ repeating-section buttons) with `cursor:pointer` + `pointer-events:auto`. `ContentControlWidgets` was the *click handler* for those painted buttons, so removing it alone leaves them **live-looking but inert** in editable docs — a regression. **A complete cut needs:** (1) the core painter to render SDTs **statically** (paint the content/value, drop the interactive trigger buttons + pointer styling); (2) remove the now-orphaned popup CSS (`.layout-sdt-widget-popup/-option/-date/-empty` in `styles/editor.css` ~952–995 + the dup in `prosemirror/editor.css`); (3) delete the React `ContentControlWidgets.tsx` + its `DocxEditor.tsx` render site; (4) drop the stale `ContentControlWidgets date input preventDefault` keep-note. **Decision needed:** confirm SDTs should render static (not interactive) — this is a core rendering change that needs a visual smoke-test, so it's left for a supervised pass. Reverted the partial cut; `phase4` keeps working content controls.
- **Document-load generation race + name/content coupling (`useDocumentLoader` + `useFileIO`, found 3g-a / review #105).** `loadBuffer` bumps `loadGenerationRef` before its async `parseDocx` and bails if it moved on; but `loadParsedDocument` (the synchronous `initialDocument` path) doesn't bump it, so a sync load that lands *during* an in-flight buffer parse gets **clobbered** by the late parse. Bumping in `loadParsedDocument` fixes that — but then the built-in Open path (`useFileIO.handleDocxFileChange`) still calls `onDocumentNameChange(result.name)` unconditionally after `await loadBuffer(...)`, even when that buffer load *bailed*, giving a **title/body mismatch** (rare: built-in Open racing an `initialDocument` change; Patrick uses `onOpen` so it's unaffected). Clean fix: `loadBuffer` returns whether it committed, `loadParsedDocument` bumps the generation, and `useFileIO` gates `onDocumentNameChange` on the committed flag. **Decision needed:** this changes the **public `DocxEditorRef.loadDocumentBuffer`** return type (`Promise<void>` → `Promise<boolean>`) — acceptable? Needs a file-open + name smoke-test.
- **Column resize corrupts auto-width tables (pre-existing, surfaced by review #101).** `useTableResizeState` reads the table's PM `columnWidths` attr to seed the resize, but that attr is **null by default** — for freshly-inserted, auto-layout, and post-`autoFitContents` tables (core `specs.ts` default + `sizing.ts` autofit). When null, `readColumnWidths` returns null, `resizeOrigWidthsRef` keeps its stale/`{0,0}` value, and on mouse-up `commitColumnResize` writes a `{0,0}`/sparse `columnWidths` array → the dragged columns **collapse to zero width**. (Parsed `.docx` tables with a `w:tblGrid` have widths, so they resize fine — likely why the Phase-1 smoke test passed.) The right fix mirrors the **row handler's** existing DOM-estimate fallback: when `readColumnWidths`/`readColumnWidthAt` is null, **measure the two adjacent columns' rendered px widths from the DOM and convert to twips** (mind zoom + colspans), seed the refs with those, then resize normally. I reverted my naive `return false` guard (it just neutered resize on the common case). **Decision needed: confirm the DOM-fallback approach (and whether auto tables should switch to `tableLayout:'fixed'` on first resize, as `commitColumnResize` already does).** Needs interactive smoke-testing.
- **Phantom comments threaded under a tracked change (pre-existing bug, found by review #99).** `useCommentManagement.cleanOrphanedComments` only orphan-cleans comments with `parentId == null`. But `useCommentLifecycle` threads a *top-level* comment under an overlapping tracked change by setting `parentId = revisionId`. So if such a comment's anchoring mark is later removed from the doc, it is never detected as orphaned and lingers in the sidebar as a phantom with no anchor. Fixing it needs a domain decision: the orphan check must distinguish a *reply* (`parentId` = parent comment id) from a *revision-threaded top-level comment* (`parentId` = revision id) — same field, two id-spaces — likely via the `commentToRevision` map. Not fixed blind because it's attorney-critical and could delete real comments if mis-handled. **Decision needed: how should orphan-cleanup treat revision-threaded comments?**
