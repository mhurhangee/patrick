# API

The API lives in `apps/api`. It's a [Hono](https://hono.dev) HTTP server running on Bun.

## Entry point

`src/index.ts` wires everything together:

```typescript
const app = new Hono()
app.use("*", cors())
app.get("/health", (c) => c.json({ ok: true }))
app.route("/projects", projectsRouter)
app.route("/assets", assetsRouter)

export default { port: 3000, fetch: app.fetch }
```

`cors()` middleware allows the frontend (running on a different port in dev, or in a Tauri webview) to make requests.

`/health` is a simple liveness check — useful for confirming the sidecar started correctly.

## Running it

```bash
pnpm dev          # watch mode — restarts on file changes
pnpm start        # run once, no watching
pnpm build        # compile to bin/api (standalone binary)
```

## Database connection

`src/lib/db.ts` sets up the database and is imported by every route file:

```typescript
import { Database } from "bun:sqlite"
import { drizzle } from "drizzle-orm/bun-sqlite"
import * as schema from "@patrickos/db"

const dbPath = (process.env.DATABASE_URL ?? "file:../../packages/db/local.db")
  .replace(/^file:/, "")
const sqlite = new Database(dbPath)
```

`DATABASE_URL` controls which database file is used. In dev it defaults to `packages/db/local.db`. When running as a Tauri sidecar, `lib.rs` sets it to a path inside the OS app data directory.

## Routes

### Projects — `src/routes/projects.ts`

| Method | Path | What it does |
|---|---|---|
| `GET` | `/projects` | List all projects |
| `GET` | `/projects/:id` | Get a single project |
| `POST` | `/projects` | Create — body: `{ name }` |
| `PUT` | `/projects/:id` | Rename — body: `{ name }` |
| `DELETE` | `/projects/:id` | Delete project and all its assets |

The DELETE handler manually deletes assets first, then the project. SQLite enforces foreign keys (`PRAGMA foreign_keys = ON`) but there's no `ON DELETE CASCADE` on the schema, so the cascade is done in code.

### Assets — `src/routes/assets.ts`

| Method | Path | What it does |
|---|---|---|
| `GET` | `/assets?projectId=` | List assets, optionally filtered by project |
| `GET` | `/assets/:id` | Get a single asset |
| `POST` | `/assets` | Create — body: `{ projectId, title, type, kind, content?, date?, notes? }` |
| `PUT` | `/assets/:id` | Partial update — only fields present in body are changed |
| `DELETE` | `/assets/:id` | Delete a single asset |

The `PUT` handler only updates fields that are present in the request body, so you can send `{ content: "..." }` without including `title` and it won't overwrite it.

## Known gaps

- **No validation** — routes trust the request body has the right shape. For MVP this is fine; Zod + Hono's validator middleware is the clean fix when it matters.
- **No pagination** — all list endpoints return everything. Fine while data is small.
