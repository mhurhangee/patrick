# Database

The database layer lives in `packages/db`. It's a shared package — the API imports the schema from here rather than defining it inline.

## Why a separate package?

Right now only the API uses the schema. But migration tooling, seed scripts, or admin utilities would all need the same type definitions. Keeping the schema in one place means changing a column name is a one-line edit that propagates everywhere.

## Drizzle ORM

Drizzle is the library that bridges TypeScript and SQL. There are two parts:

**`drizzle-orm`** — the runtime query builder used in application code:
```typescript
const rows = await db.select().from(projects)
// generates: SELECT * FROM projects
```

**`drizzle-kit`** — a CLI tool (dev-only) for generating and running SQL migration files from your schema.

## Schema

### projects

A top-level container — a patent matter or case.

```typescript
export const projects = sqliteTable("projects", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  createdAt: int("created_at", { mode: "timestamp" }).notNull(),
  updatedAt: int("updated_at", { mode: "timestamp" }).notNull(),
})
```

### assets

Documents belonging to a project. Split into Sources (uploaded PDFs) and Artifacts (in-app rich text).

```typescript
export type AssetKind = "source" | "artifact"
export type AssetType =
  | "inventor-disclosure" | "office-action" | "patent-spec"
  | "prior-art" | "claims-draft" | "response-draft"

export const assets = sqliteTable("assets", {
  id: text("id").primaryKey(),
  projectId: text("project_id").notNull().references(() => projects.id),
  title: text("title").notNull(),
  content: text("content").notNull().default(""),
  type: text("type").$type<AssetType>().notNull().default("claims-draft"),
  kind: text("kind").$type<AssetKind>().notNull().default("artifact"),
  date: text("date").notNull().default(""),
  notes: text("notes").notNull().default(""),
  createdAt: int("created_at", { mode: "timestamp" }).notNull(),
  updatedAt: int("updated_at", { mode: "timestamp" }).notNull(),
})
```

`kind` controls how the asset is rendered: `"source"` → PDF viewer, `"artifact"` → rich text editor.

`type` is the domain classification — both sources and artifacts can be any type.

`content` stores the Tiptap JSON for artifacts (rich text editor). Currently a placeholder — editor not yet wired up.

## SQLite driver choice

The API uses **`bun:sqlite`** — SQLite built directly into Bun. This differs from `@libsql/client` (which supports Turso cloud).

The reason: `@libsql/client` ships native `.node` binaries. When Bun compiles an app into a standalone binary, it bundles JavaScript but not native binaries. So `@libsql/client` fails inside a compiled Bun binary. `bun:sqlite` is part of the Bun runtime itself — always available, including inside compiled binaries.

**Trade-off:** the self-hosted and cloud tiers need `@libsql/client` to connect to remote libSQL/Turso databases. That will be wired in later — most likely by checking `DATABASE_URL` at startup and choosing the driver accordingly.

## Schema initialisation

`apps/api/src/lib/db.ts` runs `CREATE TABLE IF NOT EXISTS` on every startup:

```typescript
sqlite.exec(`CREATE TABLE IF NOT EXISTS projects (...)`)
sqlite.exec(`CREATE TABLE IF NOT EXISTS assets (...)`)
```

`IF NOT EXISTS` makes it idempotent — safe to run on every startup. This approach only works for new tables; adding columns to existing tables requires proper migrations. For now, if the schema changes materially, the local DB can simply be deleted and it will be recreated on next startup.

## WAL mode

The API enables WAL (Write-Ahead Logging):

```typescript
sqlite.exec("PRAGMA journal_mode = WAL")
```

SQLite's default mode locks the entire file for writes. WAL allows reads to continue while a write is in progress — a cheap win that avoids "database is locked" errors.
