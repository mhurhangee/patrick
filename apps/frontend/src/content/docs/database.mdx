# Database

The database layer lives in `packages/db`. It's a shared package — the API imports the schema from here rather than defining it inline.

## Why a separate package?

Right now only the API uses the schema. But in the future, database migration tooling, seed scripts, or admin utilities would all need the same type definitions. Keeping the schema in one place means changing a column name is a one-line edit that propagates everywhere.

## Drizzle ORM

Drizzle is the library that bridges TypeScript and SQL. There are two parts:

**`drizzle-orm`** — the runtime query builder you use in application code:
```typescript
const rows = await db.select().from(projects)
// generates: SELECT * FROM projects
```

**`drizzle-kit`** — a CLI tool (dev-only) for generating and running SQL migration files from your schema.

The schema is just TypeScript — it's the authoritative definition of what's in the database:

```typescript
// packages/db/src/schema.ts
export const projects = sqliteTable("projects", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  createdAt: int("created_at", { mode: "timestamp" }).notNull(),
  updatedAt: int("updated_at", { mode: "timestamp" }).notNull(),
})
```

Drizzle reads this and knows the types of every column. If you write `projects.nonExistentColumn` anywhere in the API, TypeScript will error at compile time.

## Schema: projects and artifacts

There are two tables:

**`projects`** — a top-level container. Each project has a name and timestamps. Think of it as a matter or case.

**`artifacts`** — documents that belong to a project. Each artifact has a `project_id` foreign key, a title, and a `content` field (plain text for now, Markdown or structured JSON later). Artifacts are the things AI will help draft and edit.

The `createdAt`/`updatedAt` columns are stored as Unix timestamp integers (`INTEGER` in SQLite). Drizzle's `mode: "timestamp"` converts between JavaScript `Date` objects and those integers automatically.

## SQLite driver choice

The API uses **`bun:sqlite`** — SQLite built directly into Bun. This is different from `@libsql/client` (the libSQL client that supports Turso cloud databases).

The reason: `@libsql/client` ships native `.node` binaries (pre-compiled C++ code). When Bun compiles an app into a standalone binary, it can bundle JavaScript but not native binaries. So `@libsql/client` fails to import inside a compiled Bun binary.

`bun:sqlite` is part of the Bun runtime itself, so it's always available — including inside compiled binaries.

**Trade-off:** `bun:sqlite` is SQLite only. The self-hosted and cloud tiers need `@libsql/client` to connect to remote libSQL/Turso databases. That will need to be wired in later — most likely by checking `DATABASE_URL` at startup and choosing the driver, or by building the API differently per tier.

## Schema initialisation

In a typical app you'd run migration files (SQL scripts that add/change tables) and track which ones have run. The API doesn't do this currently. Instead, `apps/api/src/lib/db.ts` runs `CREATE TABLE IF NOT EXISTS` statements every time the server starts:

```typescript
sqlite.exec(`CREATE TABLE IF NOT EXISTS projects (...)`)
sqlite.exec(`CREATE TABLE IF NOT EXISTS artifacts (...)`)
```

`IF NOT EXISTS` makes it idempotent — safe to run on every startup, nothing breaks if the table already exists.

**The limitation:** this approach can only create tables. It can't add a new column to an existing table, rename a column, or change a data type. When the schema needs to change (migration #2 onwards), this will need to be replaced with proper migration tracking.

## WAL mode

The API enables WAL (Write-Ahead Logging) mode:

```typescript
sqlite.exec("PRAGMA journal_mode = WAL")
```

SQLite's default journal mode locks the entire database file for the duration of a write. WAL mode allows reads to continue while a write is in progress. For a local app with a single user this rarely matters, but it's a cheap win and avoids mysterious "database is locked" errors.
