# Architecture

PatrickOS is a monorepo — one git repository containing multiple apps and packages that work together.

```
patrickos/
├── apps/
│   ├── frontend/   React + Vite — the UI
│   ├── api/        Hono on Bun — the API server
│   └── desktop/    Tauri — wraps the UI as a desktop app
└── packages/
    └── db/         Drizzle schema — shared between api and future tooling
```

All JavaScript/TypeScript is managed with `pnpm` workspaces. Running `pnpm install` at the root installs dependencies for every package at once.

## Domain model

The core concepts are **Projects** and **Assets**.

A **Project** is a top-level container — think of it as a patent matter or case.

**Assets** are everything inside a project, split into two kinds:

| Kind | Name | Description |
|---|---|---|
| `source` | Source | Uploaded PDF — read-only context (prior art, office actions, disclosures) |
| `artifact` | Artifact | Rich text document created and edited in the app, by humans or AI |

Every asset also has a **Type** — its domain category: `inventor-disclosure`, `office-action`, `patent-spec`, `prior-art`, `claims-draft`, or `response-draft`. Type applies to both sources and artifacts.

## Request flow

Every action in the UI goes through the same path:

```
Frontend (React)  →  API (Hono/Bun)  →  Database (SQLite)
```

The frontend never touches the database directly — it only speaks to the API over HTTP. This means the same frontend code works whether running locally, self-hosted, or in the cloud. Only the API URL changes.

## Three deployment tiers

| Tier | Frontend | API | Database |
|---|---|---|---|
| **Local** | Tauri webview | Bun sidecar binary | SQLite file in app data dir |
| **Self-hosted** | Static site | Bun server process | libSQL / sqld server |
| **Cloud** | Static site | Bun server process | Turso (managed libSQL) |

Only the **local** tier is wired up end-to-end today.

## Why these tools?

**Bun** — runs TypeScript natively (no build step for development), has SQLite built in, and can compile an entire Node-style app into a single standalone binary. That last point is what makes the Tauri sidecar possible.

**Hono** — a small, fast HTTP framework. Works identically on Bun, Node, Deno, and edge runtimes. If the API moves to a serverless platform, the code changes very little.

**Tauri** — a Rust framework for building desktop apps where the UI is a web view. It's lighter than Electron because it uses the OS's own browser engine instead of bundling Chromium.

**Drizzle ORM** — TypeScript schema → type-safe queries → plain SQL. If you reference a column that doesn't exist, TypeScript errors at compile time.

## Root scripts

```bash
pnpm dev              # frontend + API in parallel (browser dev)
pnpm dev:desktop      # Tauri desktop dev (requires build:api first)
pnpm build:api        # compile API binary + copy to desktop binaries
pnpm typecheck        # type-check all packages
pnpm lint / check     # biome lint / typecheck + lint
```
